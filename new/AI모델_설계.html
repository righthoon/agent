약손 — 복약안전 AI 에이전트 모델 설계서

사진·음성으로 약을 식별하고, 사용자의 이전 복용약·범용금기 목록과 대조해 병용 위험과 주의 음식을 판정하는 멀티모달 에이전트. 판정의 안전 코어는 LLM이 아니라 결정론적 규칙엔진으로 두어 신뢰성을 확보한다.

입력: 사진/텍스트 · 음성 Claude · Naver Clova Speech Supabase(PostgreSQL) 분석 node = 결정론적 룰엔진

2 노드별 모델·API 설계

노드역할모델 / API입력 → 출력상태
입력1
사진/텍스트		 알약·약통·처방전 이미지 또는 타이핑된 약 이름 수집 Claude 비전
claude-sonnet-5		 이미지/텍스트 → 각인·모양·성분 후보 구현됨
입력2
음성		 "내가 먹는 약 알려줘" 등 음성 발화를 텍스트로 변환 Naver Clova Speech
(STT)		 음성 → 텍스트 외부 연동
AI Agent 입력을 받아 "어떤 약인지 + 효력" 식별. 필요 시 도구 호출 Claude (tool use)
claude-sonnet-5		 이미지/텍스트 → {제품명, 성분코드, 효능, 신뢰도} 구현됨
데이터층 ① 사용자 복약이력 ② 범용금기 목록 ③ 약-음식 표 저장 Supabase
(PostgreSQL)		 SQL 조회 → 금기쌍·음식·복약이력 신규 구축
분석 node
(안전 코어)		 식별 성분 × 복약이력 × 금기목록 대조로 위험 판정 결정론적 규칙엔진
(LLM 아님)		 성분코드 → {병용금기 여부, 사유, 주의음식} 구현됨
결과 판정 결과를 어르신도 이해하게 설명 + 안전문구 Claude
(설명 생성)		 구조화 결과 → 자연어 안내 구현됨
핵심 설계 원칙. "위험한지"를 판단하는 분석 node는 절대 LLM에게 맡기지 않습니다. LLM은 (1) 입력을 약 이름으로 바꾸고 (2) 확정된 결과를 쉽게 설명할 뿐이고, 위험 판정 자체는 Supabase의 공식 금기 데이터를 대조하는 결정론적 코드가 합니다. 지어내기(hallucination)를 원천 차단하는 구조라 기술심층·윤리 배점에 유리합니다.

3 분석 node 상세 (flow.md가 요청한 부분)

flow.md의 "분석 node 설명"과 "결과 출력 항목 2가지"를 그대로 구현한 로직입니다.

# 입력: AI Agent가 식별한 성분코드(new_code) + user_id def analyze(new_code, user_id): # ① 사용자의 이전 복용약 성분들을 DB에서 조회 my_codes = db.query("SELECT ingredient_code FROM user_medications WHERE user_id=?", user_id) # ② 새 약 × 각 복용약을 '범용금기 목록'과 대조 (결정론적) risks = [] for code in my_codes: row = db.query("SELECT reason FROM dur_contraindications WHERE {new_code,code} 쌍") if row: risks.append({"상대약": code, "사유": row.reason}) # ③ 약-음식 상호작용 조회 foods = db.query("SELECT food, reason FROM drug_food WHERE ingredient=?", new_code) return { "위험판단": "위험" if risks else "현재 금기 없음", # 결과항목 1 "병용금기": risks, # 이전 복용약과의 위험 "주의음식": foods, # 결과항목 2 }

결과 출력 항목 1 — 병용 위험 판단

  • 범용금기 목록 + 이전 복용약을 대조
  • 같이 먹으면 위험한 쌍이 있으면 위험 + 사유
  • 없으면 "현재 금기 없음"으로 명확히
  • 이미 구현: check_interactions()

결과 출력 항목 2 — 주의 음식

  • 이 약과 함께 먹으면 안 좋은 음식 안내
  • 예: 와파린↔비타민K, 스타틴·혈압약↔자몽
  • 근거 불확실하면 단정 금지 → 약사 상담
  • 이미 구현: check_food()

4 Supabase(PostgreSQL) 데이터 스키마

현재 로컬 CSV(병용금기 15만행)를 Supabase 테이블로 이관합니다. 사용자 복약이력은 앱에서 누적됩니다.

테이블주요 컬럼용도

dur_contraindicationsingredient_code_1, ingredient_code_2, reason범용금기 목록(DUR)
drug_foodingredient_keyword, food, reason약-음식 상호작용
drug_masterproduct_name, ingredient_code, ingredient_name, efficacy제품명↔성분↔효력 매핑

5 AI 모델 선택 근거

왜 Claude인가 (입력1 · AI Agent · 결과)

  • 한글 처방전·알약 각인 인식(비전)이 강함
  • 도구 호출(tool use)로 결정론적 함수와 자연스럽게 결합
  • 어르신 눈높이 설명 생성에 적합
  • 기본: claude-sonnet-5(속도·비용) / 어려운 식별은 claude-opus-4-8 폴백

왜 Naver Clova Speech인가 (입력2)

  • 한국어 음성인식 정확도·발음 대응 우수
  • 어르신·비문해 사용자의 접근성 확보(SDOH 대응)
  • STT 결과 텍스트를 그대로 AI Agent로 전달

7 안전·윤리 설계 & 유료 사용 (PPT 필수)

안전 · 윤리

  • 위험 판정은 결정론적 코어가 담당 → 지어내기 차단
  • 근거 불확실 시 단정 금지 → 약사 상담 권고
  • 모든 결과 끝에 "정확한 판단은 약사·의사와 상담"
  • 의료행위가 아닌 정보 제공 명시, 개인정보 최소수집

유료 API · 인프라 (아키텍처 페이지용)

  • Claude API (비전·에이전트·설명 생성)
  • Naver Clova Speech (음성 STT)
  • Supabase (PostgreSQL 호스팅)
  • GPU: 로컬 학습 없음(추론은 API) — 비용 효율적
